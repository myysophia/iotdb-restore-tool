# IoTDB 恢复窗口内查询先无数据后恢复的排查记录

**日期**: 2026-03-25  
**问题类型**: 恢复执行中的业务查询结果波动  
**状态**: 已定位，属于恢复窗口内的预期现象

---

## 一、问题描述

业务侧反馈同一条 SQL 在短时间内出现如下现象：

```sql
select * from root.energy.sn25627A000014596.BATTERY where time >= 1773532800000;
```

表现为：

- 刚开始查询时没有数据
- 过一段时间再次查询，同样条件又能查到数据

这容易让人怀疑：

- 数据是否丢失
- 查询是否不稳定
- DataNode 是否存在异常

---

## 二、时间条件含义

SQL 中的时间戳：

- `1773532800000`

换算后是：

- **2026-03-15 00:00:00 UTC**

因此该 SQL 本身是在查历史数据，不是查未来时间，也不是条件写错导致无结果。

---

## 三、实际排查结果

### 3.1 DataNode 查询审计日志中未命中该 SQL

排查时检查了 DataNode 内的以下日志：

- `log_datanode_query_debug.log`
- `log_datanode_slow_sql.log`

未直接命中：

- 设备路径 `root.energy.sn25627A000014596.BATTERY`
- 时间戳 `1773532800000`
- 该完整 SQL

因此无法从查询日志里精确还原“该语句执行的具体秒级时间”。

### 3.2 恢复日志明确显示：查询时段内正在执行自动恢复

恢复工具日志显示，2026-03-25 当时存在一轮自动恢复任务：

| 时间（UTC） | 事件 |
|---|---|
| 2026-03-25 02:43:03 | 开始执行恢复 |
| 2026-03-25 02:43:03 | 步骤 0：删除现有数据库并清理旧数据 |
| 2026-03-25 02:43:06 | `root.energy` 删除成功 |
| 2026-03-25 03:14:24 | 所有文件导入完成 |
| 2026-03-25 03:14:26 | 数据库写读探测成功 |
| 2026-03-25 03:14:26 | 恢复操作完成 |

也就是说：

- 从 **02:43:06 UTC** 开始，`root.energy` 已被删除
- 到 **03:14:24 UTC** 前，数据一直处于重新导入中

### 3.3 系统中确实配置了周期性自动恢复

系统定时任务中存在如下恢复任务：

```text
43 */2 * * * /opt/vnnox-configcenter-data/iotdb-restore-tool/bin/iotdb-restore restore ...
```

含义是：

- 每 2 小时的 `:43` 会触发一次恢复

因此只要业务查询时间落在恢复窗口内，就可能看到短暂空结果。

---

## 四、根因结论

本次“先查不到、后面又查得到”的原因不是：

- 数据损坏
- 查询语法错误
- Kubernetes 网络抖动
- DataNode 查询引擎异常

而是：

> 查询发生在自动恢复执行窗口内。恢复任务会先删除 `root.energy`，再逐批导入 tsfile。在导入完成前，目标设备的数据可能暂时不可见；待相关文件导入完成后，同一条 SQL 就会恢复有结果。

---

## 五、为什么会出现“部分时段为空”

恢复流程当前是：

1. 删除数据库
2. 清理 live data
3. 重启 Pod
4. 重建数据库与 Region
5. 逐批导入 tsfile
6. 完成后做写读探测

在这个过程中，数据库可见性并不是“一次性恢复”：

- 删除数据库后，会出现短暂全空期
- 导入过程中，哪个 tsfile 先完成，哪个设备的数据先可见
- 某个具体设备路径的数据，在导入到对应文件前可能暂时查询不到

因此恢复窗口内可能出现三种情况：

1. 完全无数据
2. 只有部分设备有数据
3. 同一设备前一次查不到，后一次查得到

这三种都属于恢复中的正常现象。

---

## 六、为什么这次判断是成立的

虽然没有查到该 SQL 的精确执行日志，但证据链已经足够闭合：

1. 查询条件本身合法，时间范围正确
2. 恢复日志确认当时确实在执行删库和重导
3. 恢复完成前后存在明确分界点：
   - `03:14:24 UTC` 文件导入完成
   - `03:14:26 UTC` 探测成功
4. 查询“先空后有”的现象与恢复窗口行为完全一致

因此可以认定，本次现象是恢复窗口导致的数据可见性波动，而不是新的数据库故障。

---

## 七、影响说明

### 7.1 对业务的影响

- 恢复执行期间，业务查询结果可能不稳定
- 相同 SQL 在不同时间点可能返回不同结果
- 若不了解恢复窗口，容易误判为数据库故障或数据丢失

### 7.2 对运维的影响

- 运维侧如果只看单次查询结果，容易产生误报警
- 必须结合恢复任务执行时间判断结果是否可信

---

## 八、后续建议

### 8.1 验收口径调整

恢复期间不要用业务 SQL 的临时空结果作为故障结论。  
应以以下时间点作为恢复完成判断依据：

- `所有文件导入完成`
- `数据库探测成功`

只有在这两个日志出现后，业务查询结果才适合作为最终验收依据。

### 8.2 运维提示

建议后续在恢复任务执行期间增加“恢复进行中”标记，例如：

- 写本地状态文件
- 写 Kubernetes annotation
- 输出固定日志标识

这样在业务侧或运维侧看到空查询时，可以第一时间判断是否正处于恢复窗口。

### 8.3 调度优化

若业务对白天查询连续性要求高，建议调整恢复任务窗口，避免与业务查询高峰重叠。

### 8.4 文档和说明

建议把下面这条规则写入恢复 SOP：

> 恢复过程中，数据库会先删后导，业务 SQL 可能短暂返回空结果；只有在恢复日志显示“所有文件导入完成”和“数据库探测成功”之后，查询结果才可作为最终验收依据。

---

## 九、一句话结论

> 本次 `select * from root.energy.sn25627A000014596.BATTERY where time >= 1773532800000` 先无数据后恢复，不是数据丢失，而是查询发生在自动恢复窗口内；恢复任务先删除 `root.energy`，再逐批导入 tsfile，导致该设备数据在导入完成前短暂不可见。
