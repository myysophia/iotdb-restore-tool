# Raft 在 IoTDB 中如何保证一致性

**日期**: 2026-03-23
**目标**: 用大白话和架构图说明 Raft 在 IoTDB 里的作用，重点解释 ConfigNode 场景

---

## 一、先看结论

```text
Raft 保证一致性的核心只有 3 条：

1. 只有 Leader 能接收写请求
2. 写入必须复制到“大多数节点”才算成功
3. 新 Leader 必须带着最新的已提交日志上位
```

```text
所以它不是“大家各写各的，最后再同步”，
而是“先让多数节点账本一致，再宣布这条记录正式生效”。
```

## 二、Raft 在 IoTDB 架构里的位置

### 2.1 ConfigNode 场景

```text
                         ┌────────────────────┐
                         │   Client / CLI     │
                         └─────────┬──────────┘
                                   │
                                   ▼
                    ┌──────────────────────────────┐
                    │         ConfigNode 集群       │
                    │  管节点注册 / 分区 / 路由信息  │
                    └──────────────────────────────┘
                                   │
                这里的元数据一致性由 Ratis(Raft) 保证
                                   │
         ┌─────────────────────────┼─────────────────────────┐
         ▼                         ▼                         ▼
 ┌───────────────┐         ┌───────────────┐         ┌───────────────┐
 │ ConfigNode-1  │◀──────▶ │ ConfigNode-2  │◀──────▶ │ ConfigNode-3  │
 │ Leader/Follow │         │   Follower    │         │   Follower    │
 └───────────────┘         └───────────────┘         └───────────────┘
```

### 2.2 和 DataNode / Region 的关系

```text
ConfigNode
  ├─ 管“集群元数据”
  │   例如：
  │   - 哪些节点已注册
  │   - 哪个 Region 在哪些节点上
  │   - 分区怎么分配
  │
  └─ 这些关键元数据通常通过 Raft/Ratis 保证一致

DataNode
  ├─ 真正存数据
  ├─ SchemaRegion 常见也会走 Ratis(Raft)
  └─ DataRegion 可能走 IoTConsensus 或 Ratis
```

## 三、最核心的工作方式: 先记账，再生效

```text
客户端发来一个“变更请求”
例如：
- 注册一个 DataNode
- 变更某个分区信息
- 写入某条需要共识的元数据
```

在 Raft 里，不是收到请求就马上算成功，而是这样：

```text
1. Leader 先把请求写到自己的日志
2. Leader 把这条日志复制给其他 Follower
3. 超过半数节点写成功
4. Leader 才宣布：这条正式提交(committed)
5. 所有节点按同样顺序应用这条日志
```

## 四、正常写入流程图

### 4.1 总览图

```text
写请求进入后：

Client
  │
  ▼
Leader(ConfigNode-1)
  │
  ├── 先写本地日志 entry-101
  │
  ├──────────────▶ Follower(ConfigNode-2)
  │                复制 entry-101
  │
  └──────────────▶ Follower(ConfigNode-3)
                   复制 entry-101

当 3 台里至少 2 台写成功：
Leader 才把 entry-101 标记为 committed
然后通知大家应用这条日志
```

### 4.2 时间线图

```text
Client          Leader            Follower-2        Follower-3
  │                │                  │                 │
  │ 注册请求       │                  │                 │
  ├──────────────▶ │                  │                 │
  │                │ 写 entry-101     │                 │
  │                ├────────────────▶ │ 写 entry-101    │
  │                ├──────────────────────────────────▶ │ 写 entry-101
  │                │                  │                 │
  │                │ 收到多数确认      │                 │
  │                │ entry-101 提交    │                 │
  │ ◀──────────────┤ 返回成功          │                 │
  │                │                  │                 │
  │                ├────────────────▶ │ 应用 entry-101  │
  │                └──────────────────────────────────▶ │ 应用 entry-101
```

### 4.3 大白话理解

```text
可以把 Raft 想成 3 个人一起记总账：

- 只有 Leader 这个人能接单
- Leader 接单后不能自己偷偷改账
- 必须至少再让另外 1 个人也把这条账记上
- 两个人账本一致后，这条账才正式算数
```

## 五、为什么这样就能保证一致

### 5.1 原因 1: 只有 Leader 能写

```text
          写请求
            │
            ▼
      ┌──────────┐
      │ Leader   │  可以接收写入
      └──────────┘
         ▲     ▲
         │     │
   ┌────────┐ ┌────────┐
   │Follower│ │Follower│
   │  只读  │ │  只读  │
   └────────┘ └────────┘
```

```text
好处：
不会出现 3 台机器各写各的，最后彼此打架。
```

### 5.2 原因 2: 必须多数派确认

```text
3 台节点时，多数派 = 2 台

情况 A:
Leader      成功
Follower-2  成功
Follower-3  失败

=> 2/3 成功
=> 可以提交
```

```text
情况 B:
Leader      成功
Follower-2  失败
Follower-3  失败

=> 1/3 成功
=> 不能提交
=> 这条对外不能算成功
```

```text
好处：
不会出现“只有 Leader 自己知道，别人都不知道”的假成功。
```

### 5.3 原因 3: 新 Leader 必须带着最新已提交日志

```text
旧 Leader 挂了
   │
   ▼
重新选主

谁能当新 Leader？
不是谁先喊得响谁上，
而是日志必须足够新、足够完整。
```

```text
好处：
已经被多数派确认过的记录，不会被一个“落后的旧节点”覆盖掉。
```

## 六、Leader 挂掉时会发生什么

### 6.1 心跳机制

```text
正常时：

Leader ──心跳──▶ Follower-2
Leader ──心跳──▶ Follower-3

Follower 通过持续收到心跳，知道 Leader 还活着
```

### 6.2 选主流程

```text
Leader 挂了以后：

Follower-2: “一段时间没收到心跳”
Follower-3: “我也没收到”

于是开始选举
```

```text
         请求投票
Candidate-2 ─────────────▶ Node-3
Candidate-2 ─────────────▶ Node-1(已挂)

如果 Candidate-2 拿到多数票
=> Candidate-2 成为新 Leader
```

### 6.3 图示

```text
阶段 1: 原 Leader 正常

    Leader
    /    \
   /      \
 F2        F3

阶段 2: Leader 故障

    X
   / \
  /   \
 F2   F3

阶段 3: 重新选主

F2 发起投票请求 ─────▶ F3

F2 拿到 2 票中的多数
=> F2 成为新 Leader
```

## 七、为什么不会脑裂

脑裂可以理解成：

```text
两个节点都觉得自己是 Leader，并且都对外提供写入
```

Raft 靠“多数派”避免它。

### 7.1 三节点例子

```text
节点:
A  B  C

要成为 Leader
必须至少拿到 2 票
```

### 7.2 网络分裂时

#### 场景 A: A 和 B 连通，C 断开

```text
A <────────────> B      C(断网)

A/B 这一侧有 2 票
=> 可以选出 Leader

C 只有 1 票
=> 不能当 Leader
```

#### 场景 B: A 单独断开，B 和 C 连通

```text
A(断开)      B <────────────> C

A 只有 1 票
=> 不能对外继续写

B/C 有 2 票
=> 其中一个能成为 Leader
```

### 7.3 结果

```text
同一时刻，只有“拿到多数票”的那一边能继续写。

所以不会出现：
两边都在写、各自生成不同真相。
```

## 八、放到 IoTDB 的 ConfigNode 场景里

### 8.1 DataNode 注册流程

```text
DataNode 启动
   │
   ▼
向 ConfigNode Leader 注册自己
   │
   ▼
Leader 把“节点注册信息”写入 Raft 日志
   │
   ├────────▶ Follower-2 复制
   └────────▶ Follower-3 复制
   │
   ▼
多数派确认后
   │
   ├─ 无冲突: 注册成功
   └─ 有冲突: 注册失败
```

### 8.2 对应你这次故障

```text
你这次的问题，本质上就是：

ConfigNode 的共识日志里
已经有一条旧的 DataNode 注册记录
```

```text
新 DataNode 启动时：

注册请求
   │
   ▼
ConfigNode Leader 检查当前集群元数据
   │
   ▼
发现这组 ip:port 已经存在
   │
   ├─ 如果判断成“同一个旧节点重新回来”
   │    => 接受注册
   │
   └─ 如果判断成“一个新节点和老记录冲突”
        => 拒绝注册
```

### 8.3 图示

```text
ConfigNode Raft 日志中已有：

entry-88:
  DataNodeId = 1
  host = iotdb-datanode-0.iotdb-datanode.iotdb.svc.cluster.local
  ports = 6667/10730/10740/10750/10760

新的 datanode 启动后再次注册：

register(DataNodeId=?, same host/ports)
            │
            ▼
      ConfigNode Leader
            │
            ▼
比对已提交日志中的旧记录
            │
            ├─ 一致，认定旧节点回归
            └─ 冲突，拒绝注册
```

## 九、为什么 Raft 日志这么重要

```text
因为 Raft 日志不是普通运行日志，
它更像“集群状态的正式账本”。
```

里面记录的不是随便的 debug 信息，而是这种关键变更：

- 哪个节点注册成功了
- 哪个分区被创建了
- 哪个节点被移除了
- 哪条元数据变更已经提交了

```text
所以：
只删 Pod，不删保存共识状态的 PVC，
这些“正式账本”还在。
```

这就是为什么 IoTDB 重启以后，旧的节点身份和拓扑信息还会继续影响启动行为。

## 十、用一句最白的话总结

```text
Raft 的本质就是：

先让多数节点把同一条账记下来，
再告诉外部“这件事已经正式生效”。
```

```text
在 IoTDB 里，
它主要保证的是“集群元数据的一致”，
比如节点注册、分区信息、Leader 状态，
而不是让每台机器各自凭感觉决定集群现在长什么样。
```

## 十一、排障时的实际意义

遇到 IoTDB 集群类问题时，如果现象涉及：

- DataNode 注册失败
- ConfigNode 认为有旧节点冲突
- Region 分配/路由异常
- Leader 切换后状态不对

优先要想到的就是：

```text
问题可能不在“当前 Pod”，
而在“Raft 已提交的历史元数据”。
```

也就是：

- `ConfigNode` PVC 里的共识日志
- `DataNode` PVC 里的本地节点身份文件

这两边是否仍然一致。
