## 一、汇报摘要

2026-03-22 至 2026-03-23 期间，IoTDB 恢复任务出现了两类问题：

1. 恢复导入大量失败，最初一次恢复仅成功导入 `186 / 744` 个文件。
2. 后续排查中发现，`DataNode` 并非 Kubernetes 层异常，而是启动时读到了旧集群的持久化元数据，导致去连接错误的 `ConfigNode`，进而卡死在启动阶段。

最终结论如下：

- **Kubernetes 不是根因**，`Pod Running/Ready` 只是表象，不能代表 IoTDB 服务真正可用。
- **核心根因是恢复工具的解压路径被改到了根目录 `/`**，导致备份包里的旧 `system.properties` 覆盖了当前 `DataNode` 的本地持久化元数据。
- **SchemaRegion 缺失** 是另一条独立但同时存在的问题，原因是恢复流程在导入前没有确保数据库和 Region 已就绪。
- 目前已完成修复，并使用备份时间戳 **`20260323103502`** 实际恢复验证成功：
  - 成功导入 `612 / 612`
  - 失败 `0`
  - 恢复后写入和查询自检成功
  - 企微通知成功发送

### 2.1 直接影响

- 恢复任务不稳定，首次导入失败率高。
- `DataNode` 可能表现为容器存活，但数据库服务不可用。
- 若在错误流程下继续恢复，可能把旧环境的节点身份元数据带入当前集群。

### 2.2 风险级别

- **高**

原因：

- 该问题直接影响恢复成功率。
- 错误元数据可能导致节点连错集群。
- 监控层面容易被 `Pod Running` 误导，延长故障识别时间。

## 五、根因分析

### 5.1 根因一：恢复工具错误地把备份解压到了 `/`

为降低磁盘压力，恢复工具曾做过一次调整：将备份包解压目标从数据目录改到根目录 `/`。  
这个调整本意是避免把解压数据直接写进 live data 目录，但实际带来了副作用：

- 备份包中包含旧的 `/iotdb/data/datanode/system/system.properties`
- 解压到 `/` 后，该文件会直接覆盖当前运行节点的持久化元数据

这是本次问题最关键的转折点。

### 5.2 根因二：备份中包含旧集群的节点身份信息

被覆盖的 `system.properties` 中保留了旧环境信息，包括：

- 旧 `ConfigNode` 地址
- 旧命名空间域名
- 旧节点内部地址
- 旧版本信息（如 `1.1.2`）

因此 `DataNode` 重启后不会再按当前环境变量加入当前集群，而是优先使用本地持久化的旧 `config_node_list`。

### 5.3 根因三：DataNode 启动优先读取本地持久化状态

IoTDB 的行为是：

- 首次加入集群时会使用外部配置
- 之后重启更依赖本地持久化元数据

因此一旦 `system.properties` 被错误覆盖，后续重启就会稳定连错地方。

### 5.4 根因四：导入前未确保数据库和 Region 就绪

除旧元数据问题外，恢复流程原本还存在另一个缺陷：

- 删除数据库后立即导入
- 未确保 `root.emsplus`、`root.energy` 已重新创建
- 未确保 SchemaRegion/DataRegion 已进入 `Running`

这会直接导致导入过程中出现：

- `SchemaRegion doesn't exist`
- `Failed to get replicaSet of consensus group`

也就是用户提到的 “SchemaRegion 缺失问题”。

## 六、为什么这不是 Kubernetes 问题

本次问题看起来像是：

- Pod 重启
- Pod Ready
- 日志里有网络连接失败

但这些都不是根因。

真正原因是：

- `DataNode` 读取到了错误的本地持久化元数据
- 主动去连错了 `ConfigNode`
- 因协议结构和版本不兼容，启动卡死

因此这不是：

- Service 解析故障
- Pod 调度故障
- 探针配置故障
- Namespace 网络故障

而是一个 **恢复流程引入的持久化元数据污染问题**。

---

## 七、为什么“先删数据再解压”本身不是问题

中间阶段有一个怀疑点：既然恢复前已经删除数据库，而且备份包是先拷贝到 `/tmp`，为什么还会导致 `DataNode` 磁盘只读或异常。

复盘后确认：

- **“先删 live data，再解压”本身不是问题**
- 问题在于 **解压到了 `/`，覆盖了 live `system` 元数据**

只要满足下面两个条件，流程是安全的：

1. 只清理 live 数据目录 `/iotdb/data/datanode/data`
2. 备份解压到 `data_dir` 下的隔离路径，而不是 `/`

这也是最终修复方案采用的方式。

---

## 八、修复方案

本次一共落地了 6 类修复。

### 8.1 回退解压逻辑，避免覆盖 live 元数据

恢复工具已改为：

- **只删除** `/iotdb/data/datanode/data`
- **不删除** 整个 `/iotdb/data/datanode`
- 备份重新解压到 `/iotdb/data`
- 导入扫描路径使用嵌套目录：
  - `/iotdb/data/iotdb/data/datanode/data`

这样：

- 备份里的历史 `system` 文件不会覆盖 live `system`
- 导入读取的是解压出来的 tsfile
- 当前运行节点的持久化身份保持不变

### 8.2 删除旧数据后强制重启 Pod

流程调整为：

1. 删除数据库
2. 清理 live data 目录
3. 删除 `DataNode Pod`
4. 等待 StatefulSet 拉起新 Pod，并确认新 UID Ready

这样可以确保：

- 数据目录清理后的节点以干净进程重新启动
- 不在旧进程上下文里继续恢复

### 8.3 导入前主动创建数据库并等待 Region 就绪

恢复工具新增了导入前自检逻辑：

- 检查并创建 `root.emsplus`
- 检查并创建 `root.energy`
- 执行 bootstrap 写入，强制触发 Region 创建
- 轮询 `show schema regions` / `show data regions`
- 直到所有目标库的 Schema/Data Region 都是 `Running`

### 8.4 对 Region 类导入错误增加重试

当导入遇到下面这类问题时：

- `SchemaRegion doesn't exist`
- `Failed to get replicaSet of consensus group`
- `consensus group ...`

工具会自动：

1. 重新检查 Region 就绪状态
2. 等待
3. 重试导入

### 8.5 恢复完成后增加真实写读探测

在发送恢复成功通知前，工具会自动：

1. 创建探测时序
2. 实际插入一条数据
3. 立即按时间戳查询
4. 只有写入与查询都成功，才视为恢复完成

这一步解决了“文件导入完成，但数据库不一定真正可用”的问题。

### 8.6 CLI 支持 `--version`

为了让线上排障能明确当前运行版本，CLI 已支持：

- `iotdb-restore --version`
- `iotdb-restore version`

同时构建脚本已支持注入：

- 版本号
- commit
- build time

## 十一、经验总结

### 11.1 技术层面

- 对 IoTDB 这类带强持久化身份的系统，`Pod Ready` 不能代表服务真正可用。
- 恢复工具不能只关注 “文件能否导入”，必须验证 “数据库能否读写”。
- 恢复流程中对解压路径的修改，风险不亚于删除逻辑本身，尤其当备份包内包含系统级元数据时。

### 11.2 流程层面

- 任何“为降低磁盘占用而改动目录结构”的优化，都必须先验证是否会覆盖 live 持久化状态。
- 恢复类工具上线前，必须至少做一次真实环境的全链路恢复演练。
- 成功通知必须建立在业务可用性验证之上，而不是单纯建立在进程或文件统计之上。

## 十二、后续建议

### 12.1 短期建议

- 将当前恢复流程作为标准流程固化。
- 把“恢复后写读探测”设为不可关闭的默认步骤。
- 把“等待 Region 就绪”设为导入前硬性门槛。

### 12.2 中期建议

- 为恢复工具增加“只检查不执行”的预检报告，提前输出：
  - live data 目录大小
  - 备份包大小
  - 解压目标目录
  - 预计磁盘占用峰值
- 为 DataNode 启动增加更明确的健康检查，而不是仅依赖容器状态。

### 12.3 长期建议

- 梳理并标准化 IoTDB 的“恢复包内容边界”，明确：
  - 哪些目录允许恢复
  - 哪些系统元数据文件绝不能回写到线上 live 节点
- 建立恢复演练制度，要求每次关键变更后至少完成一次真实恢复回归。
